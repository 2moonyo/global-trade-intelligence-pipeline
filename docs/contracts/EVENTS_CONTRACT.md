## Status

- Event silver generation is implemented in `ingest/events/events_silver.py`
- The canonical event seed is `data/seed/events/events_seed_extended_2015.csv`
- GCS publish is implemented in `warehouse/publish_events_to_gcs.py`

## Source Systems

- Owner-curated event seed: `data/seed/events/events_seed_extended_2015.csv`
- Event bridge blueprints embedded in `ingest/events/events_silver.py`
- Generated event silver files under `data/silver/events/*`

## Purpose

- Represent disruptive events and their affected chokepoints or non-core maritime locations
- Support event-window analytics and event impact reporting
- Provide historical event context for structural vulnerability scoring
- Create a future path for user-added events and custom analytical scenarios

## Contract Decisions

- The active event contract starts from a versioned curated seed, then generated silver.
- `event_id` is the stable primary identifier across the entire event subgraph.
- Manual authoring is acceptable now, but only through the canonical extended seed and generator.
- Event geography is normalized downstream through `stg_event_location`, even though the non-core raw bridge still carries the legacy source column name `chokepoint_name`.
- Chokepoint-linked events use the shared canonical chokepoint mapping before deriving analytics `chokepoint_id`.

## Event Modelling Logic

`stg_event_location` is the canonical normalization layer:

- core chokepoints become `location_type = 'chokepoint'`
- non-core locations become normalized `location_type` values such as:
  - `port`
  - `coastal_region`
  - `maritime_passage`
  - `maritime_region`

`dim_event` derives:

- `severity_level` from `base_severity_score`
- `event_scope_type` from the observed location coverage

`bridge_event_month`:

- deduplicates event-months
- preserves lead, active, and lag flags
- aligns event windows to the conformed monthly time dimension when possible

`bridge_event_chokepoint`:

- uses canonical chokepoint labels before deriving the analytics `chokepoint_id`
- keeps event, PortWatch, and route joins consistent without depending on raw source ids

## User-Authored Future Contract

This dataset remains the most likely candidate for a future user-input workflow.

Future contract requirements should include:

- user-supplied `event_id` or deterministic generated ids
- required date range validation
- required scope typing
- required link-role semantics
- versioned edits rather than silent overwrite
- validation checks before downstream refresh

Recommended future interface:

- a bronze-style append-only event submission layer
- a validated silver event dimension and bridge builder
- preserved author metadata and submission timestamp

## Downstream Use

- `dim_event`, `bridge_event_month`, `bridge_event_chokepoint`, `bridge_event_region`, and `bridge_event_location` expose conformed event relationships.
- `mart_event_impact` combines event windows with PortWatch stress and trade exposure.
- `mart_chokepoint_monthly_stress` and related semantic marts use active event overlay fields.
- `mart_reporter_structural_vulnerability` uses historical event exposure as one structural risk input.

## Known Gaps

- Events are curated, not externally API-sourced.
- The non-core bridge source column remains named `chokepoint_name` for compatibility even when it represents a region or passage.
- User-authored event lifecycle, moderation, and versioning are not implemented yet.
